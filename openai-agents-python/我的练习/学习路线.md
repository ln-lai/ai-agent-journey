---

## 📌 学习心法（先看这个）

### 三条铁律

1. **先理解概念，再看代码**
   每个例子先问自己"它解决了什么问题？"再去看怎么写。

2. **每个阶段必须自己改一遍**
   光看不算懂。改成自己的场景跑通，才算掌握。

3. **每个阶段写一段"我学到了什么"**
   用大白话总结。说不清楚 = 没真懂。

### 评估"真的懂了"的标准

| 段位 | 表现 |
|------|------|
| 🥉 会跑 | 复制代码能运行 |
| 🥈 会改 | 改成自己的场景能跑通 |
| 🥇 会讲 | 不看代码能讲清楚原理 |
| 💎 会设计 | 给一个新需求，能从零设计 Agent 方案 |

**目标：每阶段至少到 🥇，整体到 💎**。

## 🥇 阶段 1：单 Agent 基础（3 天）

### 核心问题
**Agent 和"调用一次 GPT API" 有什么区别？**

### 必看 examples
| 文件 | 学什么 |
|------|--------|
| `examples/basic/hello_world.py` | Agent 的最小结构 |
| `examples/basic/dynamic_system_prompt.py` | 动态 system prompt（按情况切换人格）|
| `examples/basic/prompt_template.py` | 提示词模板化 |
| `examples/basic/stream_text.py` | 流式输出（边生成边显示）|

### 必做练习
- [x] ✅ 已完成：跑通 `my_hello_deepseek.py`，改成不同人格
- [x] 练习 1.1：写一个"双人格 Agent"——根据用户问的是技术问题还是情感问题，切换不同 instructions
- [x] 练习 1.2：把回答改成**流式输出**（你能看到字一个个出现）

### 升华问题（每个都要能答上来）
1. Agent 的 `instructions` 跟 ChatGPT 的 "Custom GPT 指令"是一回事吗？
2. 为什么 SDK 把"创建 Agent"和"运行 Agent"分开（`Agent(...)` 和 `Runner.run(...)`）？

## 🥇 阶段 2：工具与多轮（4 天）

### 核心问题
**为什么 LLM 需要工具？工具调用的本质是什么？**

### 必看 examples
| 文件 | 学什么 |
|------|--------|
| `examples/basic/tools.py` | `@function_tool` 装饰器 |
| `examples/basic/tool_guardrails.py` | 工具的安全边界 |
| `examples/tools/` 整个文件夹 | 各种工具：网搜、代码执行、文件操作 |
| `examples/basic/usage_tracking.py` | 跟踪 token 消耗 |

### 必做练习
- [x] 练习 2.1：写 3 个工具（查天气 + 算数 + 查时间），让一个 Agent 都能用
- [x] 练习 2.2：故意问一个"必须连调两个工具"的问题（如"北京天气怎么样？再帮我算 10/3"），观察 Agent 的多轮行为
- [x] 练习 2.3：在工具函数里 `print` 调试信息，看清 Agent 调用顺序

### 升华问题
1. LLM 真的"执行"了你的工具函数吗？真正在执行的是谁？
2. 如果工具函数报错了，Agent 会怎么处理？（看看 SDK 怎么把错误信息传回去）
3. **为什么工具的 `docstring` 写得好不好，直接决定 Agent 用得准不准？**
4. 多个工具同时被调时，是并行还是串行？

## 💎 阶段 3：Agent 设计模式（7 天）⭐ 重点

### 核心问题
**真实生产环境的 Agent 不是"单兵作战"，而是按模式组合。**

> 这是大厂面试 Agent 岗的核心考点。每个模式都要懂"什么场景用、为什么有用"。

### 必看 examples（按顺序）
| 顺序 | 文件 | 模式名 | 解决什么问题 |
|------|------|--------|--------------|
| 1 | `agent_patterns/routing.py` | **路由** | 一个总 Agent 分流到专科 Agent |
| 2 | `agent_patterns/parallelization.py` | **并行** | 同一个问题让 3 个 Agent 同时答，选最好的 |
| 3 | `agent_patterns/llm_as_a_judge.py` | **LLM 当裁判** | 用一个 Agent 评判另一个 Agent 的输出 |
| 4 | `agent_patterns/input_guardrails.py` | **输入护栏** | 拦截恶意/无关问题 |
| 5 | `agent_patterns/output_guardrails.py` | **输出护栏** | 拦截不合规输出 |
| 6 | `agent_patterns/agents_as_tools.py` | **Agent 即工具** | 把 Agent 当函数调（核心思想！）|
| 7 | `agent_patterns/forcing_tool_use.py` | **强制工具** | 让 Agent 必须用工具，不能瞎编 |

### 必做练习
- [ ] 练习 3.1：用**路由模式**写一个"全能客服"——根据用户问退款/技术/物流，分流到 3 个专科 Agent
- [ ] 练习 3.2：用**LLM 当裁判**——一个 Agent 写代码，另一个 Agent 检查代码并打分
- [ ] 练习 3.3：给上面任何一个 Agent 加**护栏**——比如不许讨论政治话题

### 升华问题（背下来）
1. **路由 vs Agent-as-tools，本质区别是什么？**（提示：谁决定调谁）
2. 为什么要"LLM 当裁判"？单 Agent 不行吗？
3. 护栏（Guardrails）在系统里相当于什么角色？（提示：和传统软件的中间件类比）
4. 如果让你设计一个"代码 Review Agent"，你会用哪几种模式组合？

## 🥇 阶段 4：多 Agent 协作（5 天）

### 核心问题
**多个 Agent 怎么协作？谁来调度？状态怎么传递？**

### 必看 examples
| 文件 | 学什么 |
|------|--------|
| `examples/handoffs/` | **Handoff（任务交接）**：A Agent 把任务移交给 B Agent |
| `examples/memory/` | 短期记忆 vs 长期记忆 |
| `examples/customer_service/` | 完整的客服系统（融合 handoff + 工具 + 护栏） |

### 必做练习
- [ ] 练习 4.1：写一个"医院导诊"Agent——分诊后 handoff 到对应科室 Agent
- [ ] 练习 4.2：给 Agent 加**长期记忆**——记住用户上次说过的偏好

### 升华问题
1. **Handoff 和工具调用，从用户视角看一样，从系统视角看哪里不同？**
2. 多 Agent 共享状态的几种方式？各自的优劣？
3. 怎么避免 Agent 之间"踢皮球"（互相 handoff 死循环）？

## 🚀 阶段 5：实战项目（7 天）🌟 简历级别

### 核心问题
**把前面学的全部串起来，做一个能放简历的项目。**

### 选 1 个深挖（别贪多）

#### 方向 A：研究助手（推荐给算法岗）
参考 `examples/research_bot/`
- 输入一个研究问题
- Agent 自动 Google 搜索、读论文、写综述
- **技术亮点**：路由 + 工具 + 并行 + 记忆

#### 方向 B：金融分析 Agent
参考 `examples/financial_research_agent/`
- 输入一家公司
- Agent 抓财报、新闻、股价，输出投资分析
- **技术亮点**：handoff + 结构化输出 + 多源数据

#### 方向 C：自定义场景（最加分）
做一个你**真的会用**的 Agent，比如：
- 自动整理 X/Twitter 高质量内容并写日报（你已经在做相关的）
- 自动追踪 GitHub 仓库更新并总结
- 个人知识库问答 Agent

### 项目交付物
1. ✅ 可运行的代码（GitHub 仓库）
2. ✅ README（架构图 + 演示截图）
3. ✅ 一篇博客/帖子（讲清楚你做了什么、踩了什么坑、学到什么）

**目标：能在面试时讲 30 分钟不重复**。
